## 目标

把 `docs/` 下的中文 Markdown 文档，在站点侧（URL）或仓库侧（文件路径）变成英文/拼音形式（如 `已知问题` → `yizhiwenti`），并且不引入链接断裂。

## 现状判断（基于仓库结构）

该仓库是 Docusaurus（存在 `docusaurus.config.ts`、`sidebars.ts`、`.docusaurus/` 生成目录）。

## 方案选择

### 方案 A（推荐，最简洁）：保持中文文件名，仅改 Docusaurus 路由 slug

适用：你主要想要“网站 URL 变英文/拼音”，并不强制要求磁盘上的文件名也变英文。

做法：
1. 在目标文档的 front matter（文件顶部 `---` 块）里增加 `slug`。
   - 例：`docs/已知问题.md` 增加 `slug: yizhiwenti`（或更语义化如 `known-issues`）。
2. 若有多级目录，按需要设置相对/绝对 slug：
   - 相对：`slug: yizhiwenti`（拼到该文档所在目录路由下）
   - 绝对：`slug: /yizhiwenti`（直接挂到站点根下）
3. 验证 sidebar、导航、站内链接：
   - Docusaurus 的 sidebar 一般按 doc id（由路径派生）引用，不需要改成英文。
   - 若有手写的 Markdown 链接指向文件路径（`[](.../已知问题.md)`），这不会因 `slug` 自动变化；如希望链接也用英文，需要改成 Docusaurus 的“按 doc 引用”的写法（后续可选增强）。

优点：
- 不需要改文件名，不需要批量更新引用，不会影响 git 历史与本地习惯。
- 直接解决“URL 里出现中文/编码”的问题。

### 方案 B：真实重命名文件（中文 → 拼音/英文）并同步更新引用

适用：你希望“仓库里文件路径本身就都是英文/拼音”，而不仅仅是网站 URL。

做法（执行期会按仓库实际引用情况收敛）：
1. 扫描所有中文命名的 `.md` 文件，生成映射表（原路径 → 新路径）。
2. 逐个 `git mv` 重命名文件/目录（避免丢历史）。
3. 全仓库更新引用：
   - Markdown 普通链接 `[]()`、图片 `![]()` 的相对路径
   - `sidebars.ts` / 其他配置里对 doc id / path 的引用（若存在）
   - i18n 目录下的 docs 是否需要同步（当前仓库存在 `i18n/es/...`）
4. 运行站点构建/本地启动，确保无死链、无缺页。

风险点：
- 引用格式不统一（相对路径、带空格/括号、锚点 `#`）会让批量替换更容易遗漏。
- i18n 目录会放大改动范围，需要确认是否也要同步重命名。

### 方案 C：编辑器插件流（不改站点机制，靠重命名工具完成）

适用：你只想“在编辑器里做一次性批量改名”，不准备引入仓库脚本。

建议（优先级从高到低）：
- 如果你在 Obsidian 管理这些 Markdown：用社区插件 **Pinyin** 做重命名，并开启“重命名时更新链接”。
- 如果你只用 VS Code：通常要“批量重命名扩展 + 链接更新扩展”组合；VS Code 上很难保证一次性把所有 Markdown 引用都更新到位，仍建议改 Docusaurus 的 `slug`（方案 A）或上仓库脚本（方案 B）。

## 推荐结论

优先落地 **方案 A（Docusaurus `slug`）**：这是最少改动、最不容易出问题、最符合“让路径变成英文”的方式。

如果你明确要“磁盘文件名也变英文”，再走 **方案 B**，但会是一次较大的重构。

## 执行步骤（用户确认后才做）

### 执行 A
1. 找到需要英文 URL 的文档清单（先从 `docs/已知问题.md` 开始）。
2. 在每个文档 front matter 增加 `slug`，并统一 slug 规则（全小写、用 `-` 分隔、不要声调）。
3. 若仓库内存在显式文件路径链接，改为更稳的 Docusaurus 引用方式（按实际项目约定选择）。
4. 本地启动/构建验证路由与链接。

### 执行 B
1. 扫描并生成中文路径 → 拼音/英文 slug 的映射表（可先只对 `docs/` 主语言目录做）。
2. 以最小批次执行 `git mv` 重命名（先 1–3 个文档验证流程）。
3. 自动替换并人工复核所有引用更新。
4. 本地启动/构建验证，修复残留引用与侧边栏配置。
